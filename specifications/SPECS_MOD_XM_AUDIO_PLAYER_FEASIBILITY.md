# MOD/XM Real-Time Audio Player Feasibility

Date: 2026-04-17

## Executive Summary

Adding real-time playback for tracker music files such as `.mod` and `.xm` is
feasible and fits HARFANG's current audio architecture well. The existing audio
mixer already streams decoded PCM into OpenAL through `IAudioStreamer`, and the
existing video-stream path already demonstrates a small C ABI loaded from a
shared library with `MakeVideoStreamer`.

The recommended target is:

- A new dynamic audio streamer loader, modeled after `MakeVideoStreamer`, but
  reusing the semantics of `IAudioStreamer`.
- A first implementation backed by `libxmp-lite`, rendering tracker modules to
  48 kHz, signed 16-bit, stereo PCM.
- High-level Lua/Python APIs generated by Fabgen, with convenience functions for
  module music playback.
- A Lua tutorial that plays `4-mat's madness.mod` from Modland when the user has
  placed the file locally.

The MVP should avoid committing the Modland module file to the repository unless
redistribution rights are explicitly cleared. The tutorial can point to the
download URL and fail with a clear message when the sample is missing.

## Feasibility Verdict

Feasible with low-to-medium integration risk.

The technical risk is moderate because the playback model is already present:
HARFANG streams WAV and OGG through a pull-based PCM interface, and tracker
libraries render directly to PCM. The main design work is deciding whether this
feature should be a built-in audio decoder or a dynamically loaded plugin like
the existing video streamer. The plugin route is slightly more work, but it
keeps third-party decoder dependencies isolated and matches the requested
ffmpeg-style model.

## Current HARFANG State

### Video Stream Plugin Model

`harfang/engine/video_stream_interface.h` defines a plain C-compatible
`IVideoStreamer` function table:

- `Startup` / `Shutdown`
- `Open`, `Play`, `Pause`, `Close`
- `Seek`, `GetDuration`, `GetTimeStamp`, `IsEnded`
- `GetFrame`, `FreeFrame`

`harfang/engine/video_stream.cpp` loads a shared library, resolves those
function names with `GetFunctionPointer`, fills an `IVideoStreamer`, and exposes
it through `MakeVideoStreamer(const char *module_path)`.

The video API is already bound in `binding/bind_harfang.py`:

- `IVideoStreamer`
- `MakeVideoStreamer`
- `IsValid`
- `UpdateTexture`

This is the model to copy conceptually.

### Audio Stream Model

`harfang/engine/audio_stream_interface.h` already defines a similar
C-compatible `IAudioStreamer`:

- `Startup` / `Shutdown`
- `Open`, `Close`
- `Seek`, `GetDuration`, `GetTimeStamp`, `IsEnded`
- `GetFrame`

The supported PCM formats are intentionally narrow:

- 44.1 kHz / 48 kHz
- signed 16-bit
- mono / stereo

`harfang/engine/audio.cpp` owns the OpenAL mixer. It periodically pulls PCM
frames from an `IAudioStreamer`, uploads them to OpenAL buffers, and already
handles looping by seeking back to zero when a stream reaches EOF.

Current built-in streamers:

- `MakeWAVFileStreamer`
- `MakeWAVAssetStreamer`
- `MakeOGGFileStreamer`
- `MakeOGGAssetStreamer`

Current script-facing APIs are hard-coded per codec:

- `StreamWAVFileStereo`
- `StreamWAVAssetStereo`
- `StreamOGGFileStereo`
- `StreamOGGAssetStereo`
- plus spatialized variants.

The missing piece is not the mixer. The missing piece is a dynamic audio
streamer loader equivalent to `MakeVideoStreamer`, plus a public scripting API
that can use that loaded audio streamer.

## Architecture Options

### Option A: Built-In Module Streamer

Add `MakeModuleFileStreamer` / `MakeModuleAssetStreamer` in the engine and
expose `StreamModuleFileStereo` / `StreamModuleAssetStereo`.

Pros:

- Smallest patch.
- Very easy Lua/Python API.
- No plugin ABI design beyond the existing `IAudioStreamer`.
- Good enough if `libxmp-lite` is an accepted always-built dependency.

Cons:

- The third-party module decoder becomes part of the core engine link.
- Less similar to the existing ffmpeg-style video streamer.
- Harder to swap playback libraries without rebuilding HARFANG.

### Option B: Dynamic Audio Streamer Plugin

Add `MakeAudioStreamer(const char *module_path)` and generic stream helpers:

```cpp
IAudioStreamer MakeAudioStreamer(const char *module_path);
bool IsValid(IAudioStreamer &streamer);

SourceRef StreamAudioFileStereo(IAudioStreamer streamer, const char *path, const StereoSourceState &state);
SourceRef StreamAudioAssetStereo(IAudioStreamer streamer, const char *name, const StereoSourceState &state);
```

Then ship an `audio_xmp` plugin that exports the `IAudioStreamer` functions.

Pros:

- Closest match to the ffmpeg video streamer.
- Keeps `libxmp-lite` or `libopenmpt` out of the core engine.
- Allows alternate module backends later.
- Keeps GPL/LGPL-sensitive codecs outside the core if another backend is ever
  tried.

Cons:

- More packaging work for Lua, Python wheels, and SDK installs.
- Requires binding `IAudioStreamer` or providing higher-level helpers.
- Requires careful plugin binary placement next to `harfang` modules and
  launchers.

### Option C: Full Music Module Object API

Expose a new object type such as `ModulePlayer`, with tracker-specific controls
for order, row, subsong, mute channels, tempo, pattern data, and metadata.

Pros:

- Best long-term feature surface for demoscene/music tooling.
- Allows visualizers and tracker debugging tools.

Cons:

- Too large for the first implementation.
- Tracker metadata differs between libraries.
- Does not add much value for the initial "play a module as background music"
  target.

Recommendation: implement Option B first, but add small convenience wrappers so
scripts do not need to manipulate raw streamers for the common case.

## Proposed C Interface

The least disruptive implementation is to keep `IAudioStreamer` as the runtime
contract and add a loader similar to `MakeVideoStreamer`.

New files:

- `harfang/engine/audio_stream.h`
- `harfang/engine/audio_stream.cpp`

Proposed loader behavior:

- Use `LoadSharedLibrary` and `GetFunctionPointer`.
- Resolve the existing `IAudioStreamer` symbol set.
- Cache shared libraries like `video_stream.cpp`.
- Treat missing required symbols as an invalid streamer.

The existing generic export names used by the video streamer (`Startup`,
`Shutdown`, `Open`, etc.) would work. A new audio plugin can also use prefixed
symbols such as `AudioStartup` if the loader is written that way, but the
current HARFANG video precedent favors the short names.

For ABI clarity, the plugin header should define an API version:

```cpp
#define HG_AUDIO_STREAM_API_VERSION 1

AUDIO_STREAM_API int GetAudioStreamAPIVersion();
AUDIO_STREAM_API int Startup();
AUDIO_STREAM_API void Shutdown();
AUDIO_STREAM_API AudioStreamRef Open(const char *name);
AUDIO_STREAM_API int Close(AudioStreamRef ref);
AUDIO_STREAM_API int Seek(AudioStreamRef ref, AudioTimestamp t);
AUDIO_STREAM_API AudioTimestamp GetDuration(AudioStreamRef ref);
AUDIO_STREAM_API AudioTimestamp GetTimeStamp(AudioStreamRef ref);
AUDIO_STREAM_API int IsEnded(AudioStreamRef ref);
AUDIO_STREAM_API int GetFrame(AudioStreamRef ref, uintptr_t *data, int *size, AudioFrameFormat *format);
```

`GetAudioStreamAPIVersion` can be optional for compatibility with the existing
`IAudioStreamer` shape, but requiring it for new dynamic audio plugins would
make diagnostics cleaner.

No `FreeFrame` is required for the MVP because the current audio mixer copies
the returned data into OpenAL immediately. The plugin can return a pointer to a
per-stream scratch buffer that remains valid until the next `GetFrame` or
`Close` call.

## Recommended Playback Library

### Primary Choice: libxmp-lite

`libxmp-lite` is the best fit for the first implementation.

Reasons:

- It is explicitly intended for games and small or embedded applications.
- It supports the target formats: MOD, S3M, XM, and IT.
- It keeps API compatibility with libxmp.
- It is MIT-licensed.
- It is C, small, and easy to wrap behind a C ABI.
- It can render directly to signed 16-bit stereo PCM at a rate chosen by the
  caller, which fits `AudioFrameFormat`.

Implementation notes:

- Use 48 kHz, signed 16-bit, stereo as the default output:
  `AFF_LPCM_48KHZ_S16_Stereo`.
- Use one `xmp_context` per `AudioStreamRef`.
- Use `xmp_load_module_from_callbacks` or `xmp_load_module_from_memory` for
  HARFANG asset streams.
- Use `xmp_play_frame` and `xmp_get_frame_info`, or `xmp_play_buffer` into a
  plugin-owned scratch buffer.
- Call `xmp_scan_module` after load if reliable duration is needed.
- Implement seek with `xmp_seek_time`, converting HARFANG nanoseconds to
  libxmp milliseconds.

The repository already mentions `libxmp-lite` in `LICENSE-3RD-PARTIES`, but no
`extern/libxmp` or `extern/libxmp-lite` source tree is present in the current
checkout. The implementation should either vendor the lite source under
`extern/libxmp-lite` or add it as a submodule and update the third-party license
notice to the exact version used.

### Alternative: libxmp Full

Full libxmp is also MIT-licensed and supports many more formats and compressed
module variants. It is still a reasonable choice, but it is more than the MVP
needs if the target is only `.mod` and `.xm`.

Use full libxmp if:

- HARFANG wants broad historical tracker format support.
- Packed/compressed module files matter.
- Slightly larger footprint is acceptable.

### Alternative: libopenmpt

`libopenmpt` is a high-quality BSD-3-Clause library with C and C++ APIs. It
supports MOD, XM, S3M, IT, MPTM, and many legacy formats. It is a strong
fallback if playback fidelity and broad compatibility matter more than minimal
size.

Tradeoffs:

- Better playback accuracy for difficult tracker files.
- Larger and more complex than `libxmp-lite`.
- C++ codebase, even though a C API exists.
- May need either integer-output API usage or float-to-S16 conversion to fit the
  current HARFANG audio stream formats.

### Lower-Priority Options

- DUMB supports IT, XM, S3M, and MOD and is permissively licensed, but the main
  project is old and modern forks describe it as semi-maintained. It is not the
  best first choice.
- libmodplug is public-domain legacy code, but OpenMPT's own documentation
  notes that it lags behind libopenmpt in quality and portability. It should not
  be selected for new work.
- micromod-style players are very small, but generally target MOD only, so they
  do not satisfy the `.XM` requirement.
- SDL_mixer can play module formats through other backends, but adding SDL only
  for this would be unnecessary because HARFANG already has its own audio mixer.

## Lua/Python API

The bindings should be generated in `binding/bind_harfang.py`, like the current
audio and video APIs.

Recommended low-level API:

```lua
streamer = hg.MakeAudioStreamer("audio_xmp")
src_ref = hg.StreamAudioFileStereo(streamer, "resources/music/song.mod", hg.StereoSourceState(1, hg.SR_Loop))
```

Python would use the same generated names:

```python
streamer = hg.MakeAudioStreamer("audio_xmp")
src_ref = hg.StreamAudioFileStereo(streamer, "resources/music/song.mod", hg.StereoSourceState(1, hg.SR_Loop))
```

Recommended convenience API when HARFANG ships a default module backend:

```lua
src_ref = hg.StreamModuleFileStereo("resources/music/song.mod", hg.StereoSourceState(1, hg.SR_Loop))
```

Recommended public functions:

- `MakeAudioStreamer(module_path) -> IAudioStreamer`
- `IsValid(IAudioStreamer&) -> bool`
- `StreamAudioFileStereo(IAudioStreamer, path, StereoSourceState) -> SourceRef`
- `StreamAudioAssetStereo(IAudioStreamer, name, StereoSourceState) -> SourceRef`
- `StreamModuleFileStereo(path, StereoSourceState) -> SourceRef`
- `StreamModuleAssetStereo(name, StereoSourceState) -> SourceRef`

Optional later functions:

- `GetModuleTitle(SourceRef)` or `GetAudioStreamInfo(...)`
- `SetModulePosition(SourceRef, order)`
- `SetModuleRow(SourceRef, row)`
- `MuteModuleChannel(SourceRef, channel, muted)`
- `SetModuleTempoFactor(SourceRef, factor)`

The MVP should rely on the existing generic source controls:

- `SetSourceVolume`
- `SetSourcePanning`
- `SetSourceRepeat`
- `PauseSource`
- `StopSource`
- `GetSourceDuration`
- `GetSourceTimecode`
- `SetSourceTimecode`

## Tutorial Plan

Add a Lua tutorial:

- `tutorials/audio_stream_module_music.lua`

Expected sample path:

- `tutorials/resources/music/4-mat's madness.mod`

Source URL:

- `https://ftp.modland.com/pub/modules/Protracker/4-Mat/4-mat%27s%20madness.mod`

Tutorial behavior:

- Initialize input and audio.
- Check that the local module file exists.
- Start looping playback with `StreamModuleFileStereo` or the explicit
  `MakeAudioStreamer` path.
- Display basic keyboard controls in the console:
  - Escape: quit.
  - Space: pause/resume.
  - Left/right: seek by a few seconds.
  - Up/down: volume.
- Shut down audio and input on exit.

Sketch:

```lua
hg = require("harfang")

local music = "resources/music/4-mat's madness.mod"

hg.InputInit()
hg.AudioInit()

local state = hg.StereoSourceState(0.8, hg.SR_Loop)
local src_ref = hg.StreamModuleFileStereo(music, state)

if src_ref == hg.SRC_Invalid then
	print("Missing or unsupported module: " .. music)
	print("Download: https://ftp.modland.com/pub/modules/Protracker/4-Mat/4-mat%27s%20madness.mod")
else
	while not hg.ReadKeyboard("raw"):Key(hg.K_Escape) do
		hg.TickClock()
	end
end

hg.AudioShutdown()
hg.InputShutdown()
```

The tutorial should not download the file automatically. Keeping the user in
control avoids silently redistributing or caching third-party music with unclear
license terms.

## Implementation Plan

1. Add dynamic audio streamer support.
   - Add `audio_stream.h/cpp`.
   - Implement `MakeAudioStreamer`.
   - Add public generic stream helper functions in `audio.h/cpp`.
   - Bind `IAudioStreamer`, `MakeAudioStreamer`, `IsValid`, and stream helpers.

2. Add the XMP backend.
   - Add `extern/libxmp-lite` or a submodule.
   - Add a CMake option such as `HG_ENABLE_XMP_AUDIO`.
   - Add a plugin target, for example `audio_xmp`.
   - Implement file and asset loading through HARFANG readers or memory buffers.
   - Render 48 kHz S16 stereo frames.

3. Add convenience APIs.
   - `StreamModuleFileStereo`.
   - `StreamModuleAssetStereo`.
   - Optionally spatialized variants only if a mono render/downmix path is
     implemented; tracker music is normally stereo and should not be falsely
     advertised as spatialized.

4. Add validation assets and tests.
   - Use a tiny generated or explicitly permissive module fixture for automated
     tests.
   - Do not use the Modland file in CI unless redistribution is cleared.
   - Test open, decode at least one frame, duration, seek, and EOF/loop
     behavior.

5. Add the Lua tutorial.
   - Use the Modland URL as manual sample acquisition.
   - Keep it close to existing audio tutorial style.

6. Package the plugin.
   - Install the plugin with the C++ SDK.
   - Include it in Lua launcher bundles.
   - Include it as Python wheel package data and ensure it can be found through
     HARFANG's shared-library loading path.

## Risks And Mitigations

- Plugin discovery and packaging: dynamic libraries must be placed where
  `LoadSharedLibrary` can find them. Mitigate with install rules and package
  tests for Lua and Python.
- Loop semantics: tracker modules often contain intentional loops. Define MVP
  behavior as "stop after the first detected song loop when `SR_Once`, seek to
  zero and continue when `SR_Loop`."
- Duration accuracy: tracker duration can require scanning. Use the library's
  scan function and accept that some exotic modules may have approximate or
  loop-dependent duration.
- Seeking cost: tracker seeking may require replaying internally up to the
  requested time. Accept this for the MVP; optimize only if real tutorials or
  games expose stutters.
- Asset streaming: tracker files are usually small enough to load into memory.
  Prefer memory or callback loading over inventing a new file abstraction.
- Spatialization: stereo music should remain stereo. Add mono downmix only if
  spatialized module sources become a real requirement.
- License hygiene: `libxmp-lite` is MIT, but the test music file may not be.
  Vendor the library, not the Modland sample, unless rights are cleared.

  2 to 4 engineer-days.

MVP plugin:

- Dynamic audio loader, XMP plugin, file and asset playback, Lua/Python

  1.5 to 3 engineer-weeks.

Production-ready integration:

- Packaging for SDK/Lua/Python, CI fixtures, install validation across Windows,
  Linux, and macOS, documentation polish:
  3 to 5 engineer-weeks.

## Recommendation

Build the feature as a dynamic audio streamer plugin from the start, backed by
`libxmp-lite`.

This gives HARFANG a reusable audio plugin model, keeps the tracker dependency
out of the engine core, matches the existing ffmpeg-style video streamer, and
still allows a small script API:

```lua
src = hg.StreamModuleFileStereo("resources/music/4-mat's madness.mod", hg.StereoSourceState(1, hg.SR_Loop))
```

If the first implementation needs to be even faster, start with an in-tree
`MakeModuleFileStreamer` prototype, but keep the public API compatible with the
dynamic plugin design so it can be moved behind `MakeAudioStreamer` without
breaking Lua/Python code.
