# Static HTML Documentation Plan

## Status

Feasibility study and implementation plan. The `static-doc` branch contains the first implementation:

- `doc/doc_to_static_html.py`
- `harfang_static_docs` and `harfang_static_docs_install` CMake targets
- `rebuild_static_docs.bat`

The current generator is intentionally warn-only for stale source links. The verified Windows build reports three unresolved references from existing documentation content: `SubmitModelToForwardPipeline` and `Scene_RemoveScript`.

## Goal

Replace the current Hugo/NPM-oriented documentation build with a first-class static HTML documentation pipeline that can be rebuilt from CMake and, on Windows, from a single batch file.

Target user flow:

```bat
rebuild_static_docs.bat
```

or directly:

```bat
cmake --build ..\build\docs-static --config Release --target harfang_static_docs_install
```

The result should be a self-contained folder containing `index.html`, CSS, images, API pages, manual pages, and a local search index. It should work from a normal static web server and should also be usable through `file://` when opened from disk.

## Current State

The current source tree already contains the important inputs and most of the parsing/rendering primitives:

- `doc/doc/` contains 951 Markdown files:
  - 24 manual pages named `man.*.md`
  - 927 API symbol documentation files
- `doc/doc/tree_desc.txt` defines manual-page order.
- `doc/img/` contains documentation images.
- `doc/markdown.css` contains a reusable documentation stylesheet.
- `doc/doc_to_hugo.py` converts the internal documentation format to Hugo content.
- `doc/doc_utils/api_tools.py` loads FabGen `api.xml`.
- `doc/doc_utils/doc_tools.py` loads Markdown documentation and can generate fallback autodoc from API XML.
- `doc/doc_utils/html_tools.py` contains an older offline HTML-oriented renderer and link resolver.
- `doc/CMakeLists.txt` currently exposes `gen_api_xml`, which generates `${build}/doc/harfang/api.xml` through FabGen.

The current CMake docs path is incomplete:

- Root `CMakeLists.txt` enables `add_subdirectory(doc)` only when `HG_BUILD_DOCS=ON`.
- Root `CMakeLists.txt` also calls `find_package(Doxygen REQUIRED)` when `HG_BUILD_DOCS=ON`, but the active `doc/CMakeLists.txt` does not use Doxygen.
- `doc/CMakeLists.txt` only generates `api.xml`; the old offline docs command is commented out and references a missing `doc_to_html.py`.
- `doc_to_hugo.py` is not wired to a CMake target in this repository.
- No active NPM pipeline is present in this repository; the NPM/Hugo complexity appears to belong to the web-site publishing side, not the local Harfang source tree.

FabGen already supports the API XML part:

- `..\FABGen\bind.py --xml --out <dir>` uses `lang.xml.XMLGenerator`.
- `XMLGenerator` emits structural API data: classes, functions, enums, constants, parameters, returns, inheritance, and static/global flags.
- The XML generator currently does not embed Markdown documentation data. Documentation is joined later from `doc/doc/*.md` by UID.

Local probe results on 2026-04-16:

- Generating `api.xml` from `binding/bind_harfang.py` took about 2.1 seconds.
- Generated `api.xml` was about 412 KB.
- The XML contained 1,708 API UIDs, including 143 classes, 60 enums, 7 constants groups, and 2,176 function entries including methods and overloads.
- 883 API UIDs had matching Markdown symbol files.
- 825 XML UIDs had no matching Markdown symbol file and can use autogenerated fallback documentation.
- 44 Markdown symbol files did not match a generated XML UID. Some are probably stale names or feature-flag-dependent symbols and should be reported by the new generator.

## Feasibility Verdict

This is feasible without Hugo, NPM, or Doxygen.

Recommended shape:

1. Keep FabGen as the source of structural API truth by generating `api.xml`.
2. Keep `doc/doc/*.md` as the editable documentation source.
3. Add a repository-local Python static site generator that consumes `api.xml` plus `doc/doc`.
4. Add CMake targets that run FabGen first, then the static generator.
5. Add a Windows batch file that configures a docs-only build tree and builds/copies the static site.

This keeps the user-facing workflow as simple as a target or batch file while avoiding coupling the site renderer directly into FabGen language backends.

## Recommended Output Layout

Generate into `${build}/doc/static_html`, then optionally copy to `${install}/docs/harfang`.

```text
static_html/
  index.html
  doc.css
  search.js
  images/
    docs/
      <HG_VERSION>/
        assetc.gif
        harfang.png
  manual/
    overview.html
    requirements.html
    cpython.html
    lua.html
    ...
  api/
    cpython/
      classes.html
      functions.html
      constants.html
    lua/
      classes.html
      functions.html
      constants.html
  reports/
    unresolved_links.txt
    stale_symbol_docs.txt
```

`index.html` should either redirect to `manual/overview.html` or render the same page shell with the overview selected.

For the first implementation, keep API pages aggregated, like the current Hugo converter:

- one classes page per language
- one functions page per language
- one constants/enums page per language

Per-symbol pages can be added later, but they are not required to remove Hugo/NPM.

## Python Work

Add a new script:

```text
doc/doc_to_static_html.py
```

Proposed command line:

```bat
python doc\doc_to_static_html.py ^
  --api <build>\doc\harfang\api.xml ^
  --doc <repo>\doc\doc ^
  --img <repo>\doc\img ^
  --css <repo>\doc\markdown.css ^
  --out <build>\doc\static_html ^
  --version <HG_VERSION> ^
  --project-name Harfang
```

Core responsibilities:

1. Load API XML with `doc_utils.api_tools.load_api`.
2. Load documentation Markdown with `doc_utils.doc_tools.load_doc_folder`.
3. Read manual page order from `tree_desc.txt`.
4. Build a route table for manual pages and API pages.
5. Render Markdown and internal directives to HTML.
6. Resolve internal links such as `[man.Assets]`, `[Vec3]`, `[Scene_CreateNode]`, and method UIDs.
7. Copy images to the generated output folder.
8. Emit a plain CSS and JS asset set with no external build step.
9. Generate a search index usable without `fetch()`, so it works from `file://`.
10. Emit validation reports for unresolved links and stale docs.

### Reuse From Existing Code

Reuse these modules instead of rewriting parsers:

- `doc_utils.metadata` for `.title`, `.img`, `.tutorial`, and related directives.
- `doc_utils.doc_tools` for Markdown loading and autogenerated API fallback docs.
- `doc_utils.api_tools` for XML UID indexing and parent lookup.
- Selected pieces of `doc_utils.html_tools`, especially generated API fragments and Markdown conversion.

The existing `html_tools.default_link_target_formatter` should not be used as-is. It was written for the old offline prototype and produces incomplete links for the desired output layout. The static generator should define a new link formatter backed by an explicit route table.

### Link Routing

Build a route table before rendering:

- `man.Overview` -> `manual/overview.html`
- `man.Assets` -> `manual/assets.html`
- class UID `Vec3` -> `api/<lang>/classes.html#vec3`
- method UID `Vec3_Normalized` -> `api/<lang>/classes.html#vec3`
- global function UID `LoadTextureFromAssets` -> `api/<lang>/functions.html#loadtexturefromassets`
- enum or constants UID `BlendMode` -> `api/<lang>/constants.html#blendmode`

Manual pages can use one default language for API links, probably `cpython`, and API pages can link within their current language.

### Search

Do not depend on NPM/Lunr for the first version.

Generate `search.js` like this:

```js
window.HG_SEARCH_INDEX = [
  { title: "Vec3", href: "api/cpython/classes.html#vec3", kind: "class", text: "3-dimensional vector." }
];
```

Add a small vanilla JavaScript search UI that performs case-insensitive substring matching over title and text. This is enough for offline documentation and avoids Node entirely. A better ranked search can be added later if needed.

### Validation

The generator should write reports and print a summary:

- unresolved internal links
- Markdown files that do not map to any XML UID
- XML UIDs without Markdown docs
- broken generated local links
- missing images

Initial behavior should be warn-only because the current tree already has mismatches. Once the reports are cleaned up, add a `--strict` option and use it in CI.

## CMake Work

Add a dedicated static docs option at the root:

```cmake
option(HG_BUILD_STATIC_DOCS "Harfang: Build static HTML documentation" OFF)
```

Include `doc/` when either docs option is enabled:

```cmake
if(HG_BUILD_DOCS OR HG_BUILD_STATIC_DOCS)
  add_subdirectory(doc)
endif()
```

Gate or remove the current Doxygen requirement:

```cmake
if(HG_BUILD_DOCS_WITH_DOXYGEN)
  find_package(Doxygen REQUIRED)
endif()
```

Static HTML generation does not need Doxygen.

Extend `doc/CMakeLists.txt`:

```cmake
set(HG_API_XML "${CMAKE_CURRENT_BINARY_DIR}/harfang/api.xml")
set(HG_STATIC_DOC_OUTPUT "${CMAKE_CURRENT_BINARY_DIR}/static_html")
set(HG_STATIC_DOC_STAMP "${HG_STATIC_DOC_OUTPUT}/.stamp")

add_custom_command(
  OUTPUT "${HG_STATIC_DOC_STAMP}"
  COMMAND ${CMAKE_COMMAND} -E rm -rf "${HG_STATIC_DOC_OUTPUT}"
  COMMAND ${CMAKE_COMMAND} -E make_directory "${HG_STATIC_DOC_OUTPUT}"
  COMMAND ${Python3_EXECUTABLE}
    "${CMAKE_CURRENT_SOURCE_DIR}/doc_to_static_html.py"
    --api "${HG_API_XML}"
    --doc "${CMAKE_CURRENT_SOURCE_DIR}/doc"
    --img "${CMAKE_CURRENT_SOURCE_DIR}/img"
    --css "${CMAKE_CURRENT_SOURCE_DIR}/markdown.css"
    --out "${HG_STATIC_DOC_OUTPUT}"
    --version "${HG_VERSION}"
    --project-name "Harfang"
  COMMAND ${CMAKE_COMMAND} -E touch "${HG_STATIC_DOC_STAMP}"
  DEPENDS
    "${HG_API_XML}"
    "${CMAKE_CURRENT_SOURCE_DIR}/doc_to_static_html.py"
    ${HG_DOC_MARKDOWN_FILES}
    ${HG_DOC_UTILITY_FILES}
  WORKING_DIRECTORY "${CMAKE_CURRENT_SOURCE_DIR}"
  VERBATIM
  COMMENT "Generating Harfang static HTML documentation"
)

add_custom_target(harfang_static_docs
  DEPENDS "${HG_STATIC_DOC_STAMP}"
)

add_custom_target(harfang_static_docs_install
  COMMAND ${CMAKE_COMMAND} -E copy_directory
    "${HG_STATIC_DOC_OUTPUT}"
    "${CMAKE_INSTALL_PREFIX}/docs/harfang"
  DEPENDS harfang_static_docs
  VERBATIM
)
```

Use `file(GLOB CONFIGURE_DEPENDS ...)` for Markdown and utility Python dependencies so adding a new doc file retriggers the generator after configure.

Keep `gen_api_xml` as the first dependency. The static docs target should not rebuild bindings for Lua or CPython; it only needs FabGen XML.

Target names:

- `gen_api_xml`: current structural API XML.
- `harfang_static_docs`: generate static HTML into the build tree.
- `harfang_static_docs_install`: copy generated static HTML into `${CMAKE_INSTALL_PREFIX}/docs/harfang`.

## Batch File Work

Add:

```text
rebuild_static_docs.bat
```

Suggested behavior:

1. Resolve `REPO_DIR`, `WORK_DIR`, `BUILD_DIR`, `INSTALL_DIR`, and `FABGEN_DIR`.
2. Locate Python exactly like `rebuild_hg_lua.bat`.
3. Configure a docs-only build tree.
4. Build `harfang_static_docs_install`.
5. Print the generated `index.html` path.
6. Optionally open it when called with `--open`.

Suggested configure flags:

```bat
cmake -S "%REPO_DIR%" -B "%BUILD_DIR%" -G "%GENERATOR%" -A "%PLATFORM%" ^
  -DCMAKE_INSTALL_PREFIX="%INSTALL_DIR%" ^
  -DHG_FABGEN_PATH="%FABGEN_DIR%" ^
  -DPython3_EXECUTABLE="%PYTHON_EXE%" ^
  -DHG_BUILD_STATIC_DOCS=ON ^
  -DHG_BUILD_DOCS=OFF ^
  -DHG_BUILD_CPP_SDK=OFF ^
  -DHG_BUILD_TESTS=OFF ^
  -DHG_BUILD_HG_LUA=OFF ^
  -DHG_BUILD_HG_PYTHON=OFF ^
  -DHG_BUILD_HG_GO=OFF ^
  -DHG_BUILD_ASSETC=OFF ^
  -DHG_BUILD_ASSIMP_CONVERTER=OFF ^
  -DHG_BUILD_FBX_CONVERTER=OFF ^
  -DHG_BUILD_GLTF_IMPORTER=OFF ^
  -DHG_BUILD_GLTF_EXPORTER=OFF
```

Then:

```bat
cmake --build "%BUILD_DIR%" --config "%CONFIG%" --target harfang_static_docs_install
```

Expected output:

```text
Static docs rebuild ok.
Open: "<work>\install\docs\harfang\index.html"
```

## Handling FabGen Documentation

No FabGen change is required for the recommended implementation.

The clean pipeline is:

```text
binding/bind_harfang.py
  -> FabGen XMLGenerator
  -> api.xml
  -> doc_to_static_html.py + doc/doc/*.md
  -> static HTML
```

Changing FabGen to embed Markdown docs directly into `api.xml` is possible but not recommended for the first pass:

- it increases XML escaping complexity
- it duplicates the existing `doc/doc` source relationship
- Python/Go docstrings already consume `--doc_md_folder` directly
- the static renderer can join docs by UID without changing FabGen

If a single XML artifact becomes important later, add optional documentation attributes to `XMLGenerator` and pass `--doc_md_folder` to the XML target. Treat that as a second phase.

## Risks And Mitigations

Risk: link behavior changes compared to Hugo.

Mitigation: generate a route table first, keep a link report, and compare generated links against the existing Hugo route assumptions.

Risk: current Markdown/API UID mismatches produce unresolved links.

Mitigation: start with warn-only validation and create reports under `reports/`.

Risk: search JSON loaded by `fetch()` does not work from `file://`.

Mitigation: generate `search.js` as a script assigning `window.HG_SEARCH_INDEX`.

Risk: CMake docs target accidentally depends on large native builds.

Mitigation: keep `harfang_static_docs` dependent only on `gen_api_xml` and Python files. The batch file should disable tools, SDKs, tests, and language builds.

Risk: optional API feature flags affect generated XML.

Mitigation: document the docs build profile and pass the same `HG_BINDING_DEFINES` used for release documentation. Include the active feature set in a generated footer or report.

Risk: old Doxygen requirement blocks static docs on machines without Doxygen.

Mitigation: gate Doxygen behind a separate option or remove it from the static docs path.

## Implementation Steps

1. Add `HG_BUILD_STATIC_DOCS` and gate `add_subdirectory(doc)` on `HG_BUILD_DOCS OR HG_BUILD_STATIC_DOCS`.
2. Gate or remove the unused Doxygen requirement from the static docs path.
3. Add `doc/doc_to_static_html.py` with argument parsing, output folder cleanup, and asset copying.
4. Implement route-table generation for manual pages and API pages.
5. Reuse `doc_utils` to render manual pages and API aggregate pages to HTML.
6. Add a simple repeated HTML shell with navigation, search box, content area, and version footer.
7. Generate `search.js` with embedded search data and vanilla JS filtering.
8. Emit validation reports for unresolved links, stale Markdown files, missing docs, and broken local hrefs.
9. Add `harfang_static_docs` and `harfang_static_docs_install` targets in `doc/CMakeLists.txt`.
10. Add `rebuild_static_docs.bat`.
11. Run a docs-only configure/build on Windows.
12. Open `index.html` directly from disk and check manual navigation, API links, image paths, and search.
13. Optionally add a CI smoke target that builds docs and fails only on generator exceptions at first.

## Acceptance Criteria

- `cmake --build <build> --target harfang_static_docs_install` generates a complete static documentation folder.
- `index.html` opens from disk without a local server.
- Manual page links such as `[man.Assets]` resolve to local HTML files.
- API links such as `[Vec3]`, `[Scene_CreateNode]`, and `[UniformSetValue]` resolve to generated API anchors.
- Documentation images render from the generated output folder.
- Search works without NPM or a network request.
- The generator prints a concise summary with counts and report paths.
- Doxygen, Hugo, and NPM are not required for the static HTML path.

## Estimated Effort

MVP static docs target:

- Python static renderer: 1 to 2 days
- CMake integration: 0.5 day
- Batch wrapper: 0.25 day
- Validation reports and smoke checks: 0.5 to 1 day

Total: about 2 to 4 days for a practical first version.

Polish beyond MVP:

- per-symbol API pages
- ranked search
- responsive styling cleanup
- strict link checking in CI
- removal or deprecation of Hugo-specific scripts

## Recommendation

Implement the static generator as a Harfang-side Python tool, not as a new FabGen backend. Keep FabGen responsible for API introspection and keep site rendering in `doc/`, where the Markdown, images, CSS, and documentation-specific logic already live.

The first useful target should be:

```bat
cmake --build <build> --target harfang_static_docs_install
```

Once that is stable, `rebuild_static_docs.bat` can become the normal local workflow.
